Firewall fundamentals.

This study write-up explains how firewalls enforce network security policies: how they evaluate ingress and egress traffic, match source, destination, protocol, and port against ordered policy entries, track connection state, and apply allow, reject, or drop actions.

My goal: I am writing this as a study exercise to organize the concepts, test my understanding, and keep a reference I can review later.
CORE / DECISION

The simplest definition is that a firewall is a hardware or software security control that monitors network traffic and applies security policies to it. Those policies determine whether traffic is allowed, actively rejected, or silently dropped.

DIRECTION / INGRESS

Ingress enters the network. Egress leaves it.

The direction is named from the perspective of the network boundary the firewall is protecting. Ingress traffic crosses that boundary into the network. Egress traffic crosses the boundary on its way out.

For example, when an internal workstation requests a web page, the request is egress traffic because it leaves the internal network for the internet. The web server’s response becomes ingress traffic when it returns through the firewall. A stateful firewall can associate that response with the connection initiated by the workstation and evaluate it as return traffic for an established session.

POLICY / RULE ORDER

ACLs were my first introduction to traffic policy.

My first practical exposure to firewall logic came in school, when I started configuring access control lists on Cisco routers or Layer 3 switches. A traditional interface ACL generally provides stateless packet filtering rather than full stateful inspection, but it follows the same core policy model: inspect packet fields, find a matching policy entry, and apply a permit or deny action.

An extended ACL can match a source address, destination address, protocol, and transport port. Cisco ACL entries are evaluated from top to bottom. The first matching entry determines the action, and traffic that matches no preceding entry reaches the implicit deny at the end. The ACL must also be applied to the correct interface and direction before it can filter traffic.

# Cisco extended ACL: permit HTTPS to one server
ip access-list extended WEB_TRAFFIC
permit tcp any host 10.0.10.20 eq 443

# Apply the ACL to an interface and direction
interface GigabitEthernet0/1
ip access-group WEB_TRAFFIC in

This example permits TCP traffic from any source to the server at 10.0.10.20 on destination port 443, then applies the ACL inbound on an example interface. The correct interface and direction depend on the network topology.

Implicit deny and default deny

At the end of a Cisco ACL, an implicit deny blocks traffic that did not match any preceding entry. Well-designed firewall policies commonly follow a default-deny model, in which traffic is denied unless an explicit rule authorizes the required communication.

Policy principle

A default-deny policy reduces unintended exposure by limiting access to documented flows. It still depends on correct rule order, explicit requirements, logging, and periodic policy review.

DEPTH / INSPECTION

Different firewalls inspect traffic at different depths.

Packet filtering

Checks basic attributes such as IP address, protocol, and port. It typically introduces less inspection overhead, but provides less context than stateful or application-layer inspection.

Stateful inspection

Tracks active connections, so it can recognize traffic that belongs to a valid session instead of treating every packet as unrelated.

Proxy / application

Acts as an intermediary and can inspect application-layer traffic more deeply. That visibility can introduce more overhead.

Next-generation firewall (NGFW)

Commonly combines traditional firewall controls with capabilities such as application control, an intrusion prevention system (IPS), identity awareness, and threat intelligence.

ACTION / RESPONSE

Firewall actions: allow, reject, or drop.

ALLOW

The firewall permits the connection.

REJECT

The firewall denies the traffic and actively returns a response, such as a TCP reset or ICMP error.

DROP

The firewall silently discards the packet. The sender receives no response and will generally wait for a timeout or retransmit.

REJECT indicates that a host or network device actively denied the traffic and returned a response; it does not necessarily identify which device generated that response. DROP returns nothing, so a scanner may be unable to determine whether filtering, packet loss, or an unresponsive host caused the missing response. Depending on the scan type, the result may appear as filtered or open|filtered. Silent dropping can slow reconnaissance by forcing retransmissions, but it should not be treated as a complete security control by itself.

PLACEMENT / SEGMENTATION

Firewalls are useful in more than one place.

  • At the perimeter: between the internet and the internal network.
  • Between internal zones: to separate users, servers, finance, development, or other sensitive areas and limit lateral movement.
  • On the host: as a local software firewall that controls traffic directly on a workstation or server.

My biggest lesson here is that a perimeter firewall is not enough on its own. If an attacker is already inside, internal segmentation and host-based controls can still reduce what they can reach.

FINAL / THOUGHTS

Final thoughts.

A firewall is one layer in a defense-in-depth architecture. It can reduce unauthorized access by enforcing policies at network boundaries, limiting permitted communication paths, and generating logs that support monitoring and investigation. Those controls are important, but they do not make the firewall a complete security system. Phishing, social engineering, insider misuse, zero-day exploitation, and malicious content carried through an allowed or encrypted session can fall outside what a firewall policy is designed to stop.

The firewall can also introduce risk when its policy is poorly designed or left unmanaged. Broad source or destination ranges, unnecessary services, temporary exceptions that become permanent, incorrect rule order, and rules that no longer have an owner can quietly expand the attack surface. Logging without review may preserve evidence, but it does not produce detection or response by itself.

This is why the value is not simply the appliance or software. The value comes from the policy behind it: knowing which systems must communicate, in which direction, over which protocol and port, and under what conditions. A useful firewall policy is specific, documented, monitored, and reviewed as the environment changes. The firewall enforces that intent, but it cannot define or maintain the intent on its own.
